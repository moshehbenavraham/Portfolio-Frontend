## üìù Documentation Standards

### Markdown Formatting

All documentation follows standard Markdown formatting:

- **Headers:** Use `#` for hierarchy
- **Code blocks:** Use `` ``` `` with language specification
- **Links:** Use relative paths for internal docs
- **Lists:** Use `-` for unordered, `1.` for ordered
- **Emphasis:** `**bold**` for important terms, `*italic*` for emphasis
- **Tables:** Use for structured data comparison

### Code Examples

All code examples should:
- Be syntactically correct
- Include comments explaining complex parts
- Follow the project's coding standards
- Be tested before committing

### Maintaining Documentation

When making changes:
1. Update relevant documentation
2. Keep examples up to date
3. Add new sections as needed
4. Update the "Last updated" date
5. Link to related documentation

## ü§ù Contributing to Documentation

Documentation improvements are always welcome!

### How to Contribute

1. **Found an error?** Open an issue or submit a PR
2. **Have a suggestion?** Propose it in a discussion
3. **Missing information?** Add it and submit a PR
4. **Outdated content?** Update it and note the changes

### Documentation Style Guide

- **Be clear and concise** - Use simple language
- **Use examples** - Show, don't just tell
- **Be consistent** - Follow existing patterns
- **Stay current** - Update with code changes
- **Link generously** - Cross-reference related docs
- **Test code examples** - Ensure they work
